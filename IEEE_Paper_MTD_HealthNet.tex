\begin{abstract}
Static IP addresses in hospital networks let attackers quietly build a complete picture of the infrastructure over days or weeks. To counter this, we developed \textbf{MTD-HealthNet}---an OpenFlow~1.3 controller that keeps changing the external-facing IPs of hospital devices on a schedule. Devices are split into three zones based on how critical they are: Critical (120\,s cycle), Imaging (100\,s), and Administrative (80\,s). Each machine holds onto a fixed 10.0.0.0/24 address internally, while only its 172.16.0.0/16 address rotates. We ran experiments on a six-node Mininet setup. With rotation on, scanners could only reliably find \textbf{1 out of 6} machines---thats a \textbf{83.3\%} drop compared to a normal static network. The attacker had to put in \textbf{4.7$\times$} more effort just to keep up. Normal traffic still got through at \textbf{97.8\%}, and swapping an address took about \textbf{12.4~ms}. So the system makes scanning way harder without really hurting day-to-day hospital operations.
\end{abstract}

\section{Introduction}

Hospitals run on networked equipment. Monitors send patient vitals to dashboards. Imaging machines pull files from storage. Pharmacy systems talk to patient databases. Everything goes over the same network. One compromised device and an attacker can potentially reach all of it.

The problem is simple---IPs almost never change. A device gets an address when its first plugged in, and it keeps that address for years. So if someone scans the network once, they have a reliable map of whats there~\cite{lei2018survey, cai2016mtd}. Ransomware, insider attacks, worms---they all take advantage of this.

Firewalls and intrusion detection systems help, but they work around a fixed layout. They dont change the layout itself. If the attacker already knows where things are, they can work around most of these defenses given enough patience.

What if the layout kept changing? If IPs are different every few minutes, old scan data becomes useless. Thats the core idea behind \textbf{MTD-HealthNet}. Rushyendra et al.\ built it with four key design goals:

\begin{itemize}
    \item \textbf{Different speeds for different devices.} Not every machine should rotate at the same rate. We group them by how important they are clinically and give each group its own timer. Lifesaving equipment rotates slowly; office PCs rotate fast.
    \item \textbf{Two-layer addressing.} Machines keep a private IP that never moves. The controller just keeps swapping the public IP that others see. Apps dont need any changes.
    \item \textbf{Real data transfer tests.} We didnt just use ping to check if things work. We actually sent encrypted data between hosts and confirmed it arrived.
    \item \textbf{Measured trade-offs.} We tracked exactly how much harder scanning gets, how much extra work the attacker needs, and what it costs the hospital in terms of packet loss and delay.
\end{itemize}

\section{Related Work}

\subsection{Overview of MTD Techniques}

There are several ways to do Moving Target Defense, and they can be grouped by what layer they work at~\cite{lei2018survey}. Some change network stuff like IPs or ports. Others mess with host-level things like memory layout. Some go after the application layer by diversifying software configs. Since we care about blocking scans, network-layer changes are most relevant here.

One issue that keeps coming up is that the more aggressively you mutate things, the more you stress your own systems~\cite{cai2016mtd}. Finding the sweet spot is still an open problem, especially when the network carries medical data where delays can matter.

\subsection{Using OpenFlow for IP Shuffling}

Dishington et al.~\cite{dishington2019ipmux} showed that you can use programmable switches to swap addresses on the fly without tanking throughput. Scans became less effective. But in their setup, every host got treated the same way. That doesnt work in a hospital---you cant handle a ventilator the same way you handle a printer.

\.{Z}al et al.~\cite{zal2024lossless} looked at the packet loss that happens right when rules get swapped out. They fixed it by running both old and new rules at the same time for a brief period. We borrowed this idea for our transition logic.

\subsection{Dealing with Old Medical Devices}

Lots of hospital gear---pumps, ultrasound machines, lab equipment---cant be updated. The firmware is locked. Huang et al.~\cite{huang2021legacy} got around this by doing the address change outside the device. The machine keeps its original IP; a proxy does the translation. Our two-tier approach works on the same principle.

\subsection{What Was Missing}

After going through the literature, Siddardha and Jeeshitha found four things nobody had properly addressed:

\begin{enumerate}
    \item Everyone uses the same rotation timer for all devices, ignoring that some can handle disruption better than others.
    \item Testing is usually done with just ICMP pings. That dosnt tell you if actual data transfers survive a rotation.
    \item Nobody had put together zone-based access rules with IP rotation in a healthcare context.
    \item There were no proper benchmarks showing both the security gain and the performace cost for hospital-type networks.
\end{enumerate}

We built MTD-HealthNet to tackle each of these. Table~\ref{tab:litcomp} puts the main mutation approaches next to each other.

\section{System Architecture}

The whole thing is split into three layers by Rushyendra et al. Each one has a specific job, and they talk to each other through clean interfaces so you could swap one out without breaking the others.

\subsection{Hospital Zone Model}

Not all hospital devices are equal. ICU machines need to stay up no matter what. Radiology workstations can afford a brief hiccup. Front-desk computers are the least critical. We modeled this with three zones:

\begin{itemize}
    \item \textbf{Critical Zone} ($h_1$, $h_2$): Think patient monitors, EHR servers. Downtime here is dangerous.
    \item \textbf{Imaging Zone} ($h_3$, $h_4$): PACS stations and radiology viewers. They move big files but can wait a second or two.
    \item \textbf{Administrative Zone} ($h_5$, $h_6$): Reception, billing, staff email. A small blip goes unnoticed.
\end{itemize}

Every device has two IPs. A fixed one from the 10.0.0.0/24 space---apps bind to this and it doesnt change ever. Then theres a rotating one from 172.16.0.50--99 that other devices on the network actually see. The controller translates between these two on every packet going through the switch.

\subsection{Controller: Ryu-Based SDN Application}

Rushyendra wrote the brain of the system as a Ryu module running OpenFlow~1.3. It does six things:

\begin{itemize}
    \item \textbf{IP assignment.} When a new host boots up and broadcasts a request, the controller hands it a private IP. Basically works like DHCP.
    \item \textbf{Header translation.} Outgoing packets get their source rewritten to the current rotating IP. Incoming ones get their destination changed back to the real internal address.
    \item \textbf{Flow priorities.} ARP goes at 1000, ICMP at 500, host-specific NAT at 100, default at 1. This layering stops generic rules from overriding specific ones when a rotation is happening.
    \item \textbf{DNS updates.} Every time an IP rotates, hostname mappings get refreshed so other hosts can still find each other.
    \item \textbf{Access control.} Before any flow rule goes into the switch, the controller checks zone policies. If two zones arent supposed to talk, no rule gets installed. Period.
    \item \textbf{Logging.} Every single allocation, rotation, and policy decision goes into SQLite. Good for debugging and also for audit trails.
\end{itemize}

Theres also an HTTP API on port~8000 where operators can check status, look at logs, resolve names, or trigger a manual shuffle.

\subsection{Network Layer: Mininet Topology}

Kishan set up the virtual network in Mininet. Six hosts, one Open vSwitch, all controlled by Ryu. Each host has a small Python web server on port~8080 that can take encrypted messages and send back confirmations. Port~8888 runs a helper service that lets us execute commands on any host remotely during testing.

\subsection{Rotation Scheduler}

Rushyendra and Kishan wrote a Python script that watches the clock for each zone and hits the controller's shuffle API when a zone's time is up. All the config---which hosts go where, rotation intervals, who can talk to whom---lives in one YAML file called \texttt{policies.yml}.

Siddardha came up with the access rules in that file. The logic is: more sensitive zones get wider access. Critical can talk to anyone. Administrative is the most locked down. Rules are checked top to bottom, first match wins.

\section{Methodology}

\subsection{How Rotation Works}

Algorithm~\ref{alg:iphopping} by Rushyendra lays out the rotation steps. When enough time has passed for a zone, the scheduler goes through each host in it, picks a new IP from whats available in the pool, deletes the old flow rules, puts in new ones, updates DNS, and logs everything.

\begin{algorithm}[htbp]
\caption{Zone-Based IP Hopping Procedure}
\label{alg:iphopping}
\begin{algorithmic}[1]
\Require Zone configuration $Z = \{z_1, z_2, z_3\}$, IP pool $P$, interval map $I$
\Ensure Updated NAT mappings and flow rules
\For{each zone $z_i \in Z$}
    \If{$elapsed\_time(z_i) \geq I(z_i)$}
        \For{each host $h_j \in z_i$}
            \State $ip_{old} \gets \text{NAT}[h_j.\text{private\_ip}]$
            \State $ip_{new} \gets \text{RandomSelect}(P \setminus \text{ActiveIPs})$
            \State \text{DeleteFlows}($h_j$, $ip_{old}$)
            \State \text{InstallNATFlows}($h_j$, $ip_{new}$)
            \State \text{NAT}[$h_j.\text{private\_ip}$] $\gets ip_{new}$
            \State \text{UpdateDNS}($h_j.\text{hostname}$, $ip_{new}$)
            \State \text{LogShuffle}($h_j$, $ip_{old}$, $ip_{new}$)
        \EndFor
        \State \text{ResetTimer}($z_i$)
    \EndIf
\EndFor
\end{algorithmic}
\end{algorithm}

On the wire, two OpenFlow rules do the actual rewriting:

\begin{verbatim}
# Outbound: private_src -> public_src
match = OFPMatch(eth_type=0x0800,
                 ipv4_src=private_ip)
actions = [OFPActionSetField(
               ipv4_src=public_ip),
           OFPActionOutput(out_port)]

# Inbound: public_dst -> private_dst
match = OFPMatch(eth_type=0x0800,
                 ipv4_dst=public_ip)
actions = [OFPActionSetField(
               ipv4_dst=private_ip),
           OFPActionOutput(in_port)]
\end{verbatim}

To avoid dropping packets mid-transition, Rushyendra and Kishan keep both old and new rules active at the same time for a short window. The old ones only get deleted once the switch has confirmed the new ones are in place.

\subsection{What the Attacker Can Do}

Siddardha et al.\ laid out four things the attacker might try:

\begin{itemize}
    \item \textbf{Scanning.} Run nmap or similar to find live hosts and open ports.
    \item \textbf{Pivoting.} After getting into one machine, try to reach others in different zones.
    \item \textbf{Spoofing.} Fake the source address on packets to abuse trust between devices.
    \item \textbf{Ransomware.} Use the network map to target high-value servers with encryption malware.
\end{itemize}

We assume the attacker has no way to talk directly to the Ryu controller and cannot see the 10.0.0.0/24 internal addresses. Under these rules, the goal is to make scan results go stale before the attacker can use them, make lateral movement expensive, and limit how long each device sits at a known address.

\subsection{What We Measured}

Rushyendra and Siddardha picked four metrics:

\begin{enumerate}
    \item \textbf{Scan yield ($R_A$).} Out of all active hosts, how many does one scan find?
    \begin{equation}
        R_A = \frac{|\text{Hosts discovered with valid IPs}|}{|\text{Total active hosts}|}
    \end{equation}

    \item \textbf{Effort multiplier ($E_M$).} How many more scans does the attacker need compared to a static network?
    \begin{equation}
        E_M = \frac{\text{Scans required under MTD}}{\text{Scans required under static}} = \frac{T_{hop}^{-1} \cdot T_{scan}}{1}
    \end{equation}
    $T_{hop}$ is the rotation interval. $T_{scan}$ is how long one scan takes.

    \item \textbf{Delivery rate ($PDR$).} What percent of packets make it through during a swap?
    \begin{equation}
        PDR = \frac{\text{Packets received}}{\text{Packets sent}} \times 100\%
    \end{equation}

    \item \textbf{Swap time ($\delta_f$).} Milliseconds from the shuffle command to the new rules being live on the switch.
\end{enumerate}

\section{Implementation}

Rushyendra and Kishan put together the prototype using:

\begin{itemize}
    \item \textbf{Controller.} Python~3.8 with Ryu and eventlet~0.30.2, all in a Docker image. Talks OpenFlow~1.3 over TCP.
    \item \textbf{Virtual network.} Mininet~2.3.0 with Open vSwitch~2.13 for the switching fabric.
    \item \textbf{Host services.} Each virtual host runs a small HTTP server on 8080 that recieves encrypted payloads, checks them, and writes a receipt with a timestamp.
    \item \textbf{Scheduler.} Standalone Python script that polls time and calls the shuffle API.
    \item \textbf{Dashboard.} Simple web page (HTML/CSS/JS) that hits \texttt{/status} and \texttt{/logs} periodically and shows live mappings and rotation history.
    \item \textbf{Testing tools.} Nmap for scanning, Wireshark for packet capture, cURL for sending encrypted transfers.
\end{itemize}

Since the controller, scheduler, and HTTP server all touch the same data (host table, NAT mappings, logs), we use a mutex on every write. Everything also gets saved to SQLite in four tables: \texttt{hosts}, \texttt{nat\_mappings}, \texttt{shuffle\_log}, \texttt{dns\_records}.

For verifying that communication actually works end to end, Kishan and Jeeshitha used the \texttt{/sim/secure\_transfer} endpoint instead of \texttt{pingall}. The reason is that \texttt{pingall} sometimes gives false negatives when address rewriting messes with ICMP replies. The secure transfer endpoint does a complete encrypted send-recieve-acknowledge cycle.

\section{Experimental Setup}

\subsection{Lab Environment}

We used six Mininet hosts ($h_1$--$h_6$) connected through a single Open vSwitch under one Ryu controller. $h_1$ and $h_2$ are Critical, $h_3$ and $h_4$ are Imaging, $h_5$ and $h_6$ are Administrative. Internal IPs go from 10.0.0.1 to 10.0.0.6. The rotation pool is 172.16.0.50--99, so 50 addresses for 6 hosts. The Docker container for the controller uses \texttt{--network host} so it shares the network namespace with Mininet.

\subsection{Rotation Intervals}

Table~\ref{tab:zoneconfig} shows the timings we ended up with after some trial and error.

\begin{table}[htbp]

\label{tab:zoneconfig}

\textbf{Hospital Zone} & \textbf{Risk Level} & \textbf{Hosts} & \textbf{Interval (s)} \\
\midrule
Administrative & Low & $h_5, h_6$ & 80 \\

It looks weird that Critical rotates the slowest. But we found that when Critical devices rotated too fast, there was sometimes a gap between when the controller updated the rules and when the host daemon caught up. That gap could break active telemetry feeds. Since these machines can be keeping patients alive, we went with a longer but rock-solid cycle. Admin machines dont have that constraint, so they get the fastest rate.

\subsection{Test Scenarios}

Rushyendra et al.\ ran three types of experiments:

\begin{enumerate}
    \item \textbf{Scan accuracy decay.} We did repeated \texttt{nmap -sn 172.16.0.0/24} sweeps and watched how the attacker's view degraded over successive rotations.
    \item \textbf{Pivot attempts.} An attacker on an Administrative host would scan for a Critical target, wait a bit, then try connecting. The address might have changed in between.
    \item \textbf{Normal traffic during rotation.} Pairs of hosts sent encrypted data through the REST API during entire rotation cycles. We counted successes and failures.
\end{enumerate}

\textit{Assumptions.} (i)~Attacker cant access the controller or internal IPs. (ii)~One full scan takes roughly as long as one rotation period. (iii)~Mininet behaves similarly to real OpenFlow hardware. (iv)~Single-switch latency numbers are a best case.

\section{Results and Evaluation}

\subsection{Scans Became Much Less Useful}

Rushyendra and Kishan did 20 back-to-back scans (15--20\,s each) on both the static and rotating networks.

Static network: every scan found all six hosts. No surprises there. With MTD running, things fell apart fast:

\begin{itemize}
    \item \textbf{Scan 1}: Six hosts visible. Rotation hadnt kicked in yet.
    \item \textbf{Scans 2--5}: Around 3.2 hosts per scan on average, once Admin IPs started cycling.
    \item \textbf{Scans 6--20}: About one host per scan after all zones had rotated at least once.
\end{itemize}

The long-run number settled at \textbf{16.7\%}---basically one out of six---which is an \textbf{83.3\% drop} from perfect visibility. That one host is just whichever got its IP most recently and hasnt rotated yet.

\subsection{Attacker Has to Work Much Harder}

Without rotation, one scan gives you everything. With rotation, each scan is only good until the next cycle. For the Administrative zone (80\,s rotation, 18\,s scan time):

\begin{equation}
    E_M = \frac{80}{18} \approx 4.4\times \text{ (Administrative zone)}
\end{equation}

Critical zone with 120\,s:

\begin{equation}
    E_M = \frac{120}{18} \approx 6.7\times \text{ (Critical zone)}
\end{equation}

Weighted average across all zones is \textbf{4.7$\times$}. The attacker basically needs to scan almost five times more to know whats going on.

\subsection{Lateral Movement Mostly Fails}

Kishan ran 15 pivot tests. Each one: scan the target, pause, then try to connect.

\begin{itemize}
    \item \textbf{Static}: 15/15 worked (100\%).
    \item \textbf{With MTD}: 3/15 worked (\textbf{20\%}).
\end{itemize}

12 out of 15 failed because the IP had already moved by the time the attacker tried to connect. On top of that, the zone policy blocks Admin-to-Critical traffic anyway, so even a correct address wouldnt help in many cases.

\subsection{Performance Cost is Small}

Table~\ref{tab:perfeval} has all the numbers.

\begin{table}[htbp]
\centering
\caption{Performance Evaluation: Static vs.\ MTD-HealthNet}
\label{tab:perfeval}
\begin{tabular}{lccc}
\toprule
\textbf{Metric} & \textbf{Static} & \textbf{MTD} & \textbf{Impact} \\
\midrule
Packet Delivery Ratio & 99.8\% & 97.8\% & \textbf{2.0\% decrease} \\

What we observed (Rushyendra et al.):

\begin{itemize}
    \item That \textbf{2\% packet loss} only happened in the brief moment when rules were being swapped. The overlap window kept it from being worse.
    \item Regular RTT went up by \textbf{0.07~ms} because of header rewriting on each packet. Mid-rotation it spiked to 2.8~ms but recovered in a couple seconds.
    \item \textbf{12.4~ms} to update rules is nothing for clinical software---those apps work on a scale of seconds, not milliseconds.
    \item All encrypted transfers via the REST endpoint completed \textbf{100\%} of the time, even when a rotation was happening.
\end{itemize}

\section{Case Studies}

\subsection{Scenario A: Ransomware Going After Patient Data}

\textit{Setup.} Attacker gets into $h_5$ (Admin) and wants to hit $h_1$ (Critical, holds patient records).

\textit{On a static network.} Scan once, get $h_1$'s address, deliver ransomware. Easy.

\textit{With MTD-HealthNet.} Scan says $h_1$ is at 172.16.0.57. Building the ransomware takes 60--90 seconds. But $h_1$ rotates every 120\,s, so by the time the payload is ready, 172.16.0.57 might belong to someone else or no one at all. And even if the attacker scans again, the access policy blocks Admin-to-Critical traffic independently. Two separate defenses have to both fail for the attack to land.

\subsection{Scenario B: Moving Laterally Within Imaging}

\textit{Setup.} Attacker on $h_3$ (Imaging) wants to steal radiology files from $h_4$.

\textit{With MTD-HealthNet.} Attacker notes $h_4$ at 172.16.0.62. One Imaging cycle later (100\,s), $h_4$ is at 172.16.0.78. Connecting to the old address does nothing. The attacker can scan again, but the next rotation is already coming. They end up chasing an address that keeps moving.

\subsection{Scenario C: Non-Stop Scanning}

\textit{Setup.} Outsider runs scans continuously.

\textit{With MTD-HealthNet.} First scan gets six IPs. 80 seconds later, Admin addresses are different. At 100 seconds, Imaging ones change too. By 120 seconds, everything has cycled once. The accumulated data is a mess of current and expired entries---maybe one in six is still valid. And all the addresses come from the 172.16.0.0/16 pool anyway, so the attacker learns nothing about the actual 10.0.0.0/24 layout underneath.

\section{Discussion}

\subsection{What Works Well}

\begin{itemize}
    \item \textbf{Scan data goes stale.} After one rotation cycle, whatever the attacker found is mostly useless. Repeated scanning gives diminishing returns.
    \item \textbf{Internal network stays hidden.} Even if you know every address in the rotation pool, you still know nothing about the permanent IPs or which device is which.
    \item \textbf{Per-zone knobs.} Each ward can have its own rotation speed. The ICU gets gentle handling, guest wifi gets aggressive rotation.
    \item \textbf{Tested beyond ping.} We confirmed that encrypted payloads actually arrive, not just that ICMP works.
    \item \textbf{Full audit log.} SQLite stores every address change with timestamps. Useful for HIPAA compliance and incident investigation.
\end{itemize}

\subsection{What Needs Improvement}

\begin{itemize}
    \item \textbf{Single controller.} If the Ryu process goes down, all defenses stop. Theres no redundancy yet.
    \item \textbf{Tiny topology.} One virtual switch isnt realistic. Real hospitals have complex multi-building networks.
    \item \textbf{Firmware-locked devices.} Some equipment has hardcoded peer addresses. Wed need a translation proxy for those, which we havent built yet.
    \item \textbf{Address pool size.} 50 candidate IPs for 6 devices gives about $\log_2\binom{50}{6} \approx 22.7$ bits of entropy. Decent, but nowhere near cryptographic strength.
    \item \textbf{Cant rotate too fast.} Under about 30 seconds, we saw timing issues between rule installation and daemon updates that caused communication problems.
\end{itemize}

\subsection{Is It Worth the Cost?}

Rushyendra and Jeeshitha looked at this directly. You lose 2\% on packet delivery and add 0.07~ms of latency in steady state. In exchange you get 83.3\% less scan accuracy for attackers and 80\% fewer successful pivots. Hospital software usually cares about seconds, not sub-millisecond differences, so the overhead is basically invisible.

And because each zone has its own timer, admins can fine-tune things. A neonatal ICU could run at 180-second cycles while a public wifi segment goes at 45 seconds. Same framework handles both.

\section{Conclusion}

We wanted to find out if regularly changing IP addresses is feasible for hospital networks without causing problems. It is. MTD-HealthNet cut scan effectiveness by 83.3\%, forced attackers to scan 4.7$\times$ more often, and blocked 80\% of lateral movement attempts. Packet delivery stayed above 97.8\% and the added latency was negligible.

No special hardware needed. The overhead is low enough for clinical use. Zone-based timers give security teams the flexibility to protect each department according to its own needs.

\section{Future Work}

Jeeshitha and Siddardha suggest these next steps:

\begin{itemize}
    \item \textbf{Smart timing.} Instead of fixed intervals, adjust rotation speed based on anomaly scores. Speed up when somthing looks suspicious, slow down when its quiet.
    \item \textbf{IDS integration.} Let an intrusion detection system trigger an immediate IP swap for any host it flags. Adds a reactive layer on top of the scheduled rotation.
    \item \textbf{Bigger testbed.} Test with multiple switches and subnets. See how the controller handles more devices and more complex topologies.
    \item \textbf{Port rotation.} Change listening ports along with IPs, rewriting them at the switch so host software doesnt notice.
    \item \textbf{Game theory.} Model the attacker-defender interaction formally to find mathematically optimal rotation intervals for given scan speeds and network sizes.
    \item \textbf{Real hospital trial.} Deploy on actual hospital infrastructure to get measurements under real traffic, real latency requirements, and real compliance constraints.
\end{itemize}
