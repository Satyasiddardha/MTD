\begin{abstract}
Most hospital networks still use fixed IP addresses, which gives attackers a stable map they can study and exploit at leisure. We present \textbf{MTD-HealthNet}, a software-defined networking (SDN) framework that fights back by constantly shuffling the public-facing IPs of hospital devices. A Ryu-based OpenFlow~1.3 controller sorts hosts into three risk zones---Critical, Imaging, and Administrative---and rotates their addresses every 120, 100, and 80 seconds, respectively. Internally, every device keeps its private address unchanged, so applications never notice the shuffle; externally, the network looks different every minute. We tested the system on a Mininet testbed with six emulated hospital endpoints. The results show an \textbf{83.3\%} drop in reconnaissance accuracy (from~6/6 to~1/6 hosts found per scan), a~\textbf{4.7$\times$} increase in the effort an attacker must spend scanning, and a packet delivery ratio that stays above \textbf{97.8\%} even while IPs are changing. Flow rules update in roughly \textbf{12.4~ms} on average. These numbers suggest that dynamic address rotation can meaningfully harden hospital networks without disrupting the clinical services that depend on them.
\end{abstract}

\section{Introduction}

Hospitals today run on networks. Electronic health records, bedside monitors, imaging servers, and pharmacy dispensers all talk to each other over a shared LAN. That connectivity is essential for patient care---but it also makes hospitals attractive targets. A single compromised device can give an attacker a foothold to reach every other system on the same network.

The core weakness is predictability. Most hospital LANs assign static IPs that remain unchanged for months or years. An attacker who scans the network once can build an accurate map that stays useful indefinitely. With that map, targeted ransomware delivery, lateral movement toward EHR databases, and IP spoofing all become straightforward~\cite{lei2018survey, cai2016mtd}.

Firewalls and intrusion detection systems help, but they are reactive. They block known threats without changing what the network looks like to an attacker. Moving Target Defense (MTD) takes the opposite approach: it keeps changing the network itself, so any intelligence the attacker gathers expires quickly.

We built \textbf{MTD-HealthNet}, an SDN-based framework designed and implemented by Rushyendra et al., that applies this idea to hospital LANs. Our main contributions are:

\begin{itemize}
    \item A \textbf{zone-based rotation scheme} that assigns different IP-change intervals to Critical, Imaging, and Administrative zones, matching each zone's tolerance for disruption.
    \item A \textbf{transparent NAT layer} that keeps every host's private IP fixed while constantly shuffling the public IP it shows to the rest of the network.
    \item A \textbf{working prototype} built with Ryu, Mininet, and Open vSwitch, tested with real application-layer data transfers---not just ping.
    \item \textbf{Hard numbers} on how much reconnaissance accuracy drops, how much more work an attacker must do, and how little overhead the system adds.
\end{itemize}

\section{Related Work}

\subsection{Moving Target Defense Techniques}

Lei et al.~\cite{lei2018survey} sort MTD approaches into three broad families: network-level (changing IPs, ports, or routes), host-level (randomizing memory layouts or OS fingerprints), and application-level (diversifying software stacks). Among these, network-level randomization is especially good at defeating reconnaissance, because it directly invalidates the scan data an attacker relies on.

Cai et al.~\cite{cai2016mtd} review the field more broadly and call out a tension that runs through all MTD work: the faster you mutate, the harder life gets for attackers, but the more overhead you impose on legitimate traffic. Striking the right balance remains an open problem.

\subsection{IP Shuffling in SDN Networks}

Dishington et al.~\cite{dishington2019ipmux} show that an OpenFlow controller can multiplex IP addresses on the fly, cutting scanning success rates without hurting throughput. Their results confirm that SDN is a natural platform for network-layer MTD. However, they treat every host the same way and do not consider the mixed-criticality environment of a hospital.

\.{Z}al et al.~\cite{zal2024lossless} tackle the packet-loss problem head-on. By pre-computing replacement flow rules and briefly keeping old and new rules active at the same time (a ``shadow period''), they achieve near-zero loss during transitions. We borrow this shadow-period idea in our own flow management.

\subsection{MTD for Healthcare and Legacy Devices}

Huang et al.~\cite{huang2021legacy} focus on a practical concern: many legacy devices cannot change their own network settings. Their solution wraps address changes around the device rather than inside it. This thinking shaped our dual-layer NAT design---internal IPs stay fixed for compatibility, while external IPs rotate for security.

\subsection{What is Still Missing}

Our literature review, led by Siddardha and Jeeshitha, found four gaps that motivated this work:

\begin{enumerate}
    \item Most MTD systems rotate every host at the same rate, ignoring that a ventilator and a billing workstation have very different uptime needs.
    \item Nearly all evaluations rely on ICMP ping to check connectivity; none verify that actual application data survives an IP change.
    \item Combining policy-based access control with dynamic rotation in a healthcare setting has received little attention.
    \item Concrete measurements of the security--performance trade-off for hospital-scale topologies are scarce.
\end{enumerate}

MTD-HealthNet tackles each of these gaps. Table~\ref{tab:litcomp} summarizes how common MTD techniques compare.

\section{System Architecture}

Rushyendra et al.\ designed MTD-HealthNet around three layers, each with a clear job.

\subsection{Network Model}

We split the hospital LAN into three zones, mirroring how real hospitals segment their equipment:

\begin{itemize}
    \item \textbf{Critical Zone} ($h_1$, $h_2$): ICU monitors and EHR servers---anything whose downtime could endanger patients.
    \item \textbf{Imaging Zone} ($h_3$, $h_4$): PACS and DICOM workstations handling medical images.
    \item \textbf{Administrative Zone} ($h_5$, $h_6$): Billing terminals, staff email, and other office systems.
\end{itemize}

Every host gets two IPs. The \textit{private} one (10.0.0.x) never changes; it is what the host's own software sees. The \textit{public} one (drawn from 172.16.0.50--99) is what the rest of the network sees, and it rotates on a schedule. This way, applications keep working normally while the externally visible layout keeps shifting.

\subsection{Control Plane: Ryu SDN Controller}

The brain of the system is a custom Ryu controller, implemented by Rushyendra, running OpenFlow~1.3. It handles six key jobs:

\begin{itemize}
    \item \textbf{DHCP simulation}: Hosts start with no IP and request one from the controller, just like a real DHCP handshake.
    \item \textbf{NAT management}: For every packet leaving a host, the controller rewrites the source IP from private to public. For every packet arriving, it does the reverse.
    \item \textbf{Flow rule orchestration}: Rules are installed at four priority levels (ARP at~1000, ICMP at~500, NAT at~100, default at~1) so that higher-priority traffic is never blocked by generic rules.
    \item \textbf{DNS updates}: After each shuffle, the controller updates its DNS table so that other hosts can still find the rotated host by name.
    \item \textbf{Policy enforcement}: Before installing any flow, the controller checks zone-based ACLs to make sure the communication is allowed.
    \item \textbf{State persistence}: Every mapping, log entry, and DNS record is written to an SQLite database, so the system can recover after a restart.
\end{itemize}

A REST API on port~8000 lets administrators check status, view logs, resolve hostnames, and trigger manual shuffles.

\subsection{Data Plane: Mininet Topology}

Kishan built the data plane as a Mininet virtual network: six hosts, one Open vSwitch, and one controller. Each host also runs a small Python HTTP server on port~8080 that can receive encrypted payloads and send back acknowledgment logs. A separate topology agent on port~8888 lets us run shell commands on any virtual host during testing.

\subsection{MTD Engine: Zone Scheduler}

The zone scheduler, co-developed by Rushyendra and Kishan, is a standalone Python script that watches the clock and calls the controller's shuffle API when a zone's timer expires. Zone policies---which hosts belong where, how often they rotate, and who can talk to whom---live in a YAML file (\texttt{policies.yml}).

Access rules, designed by Siddardha, follow a simple hierarchy: Critical-zone hosts may reach any zone, Imaging-zone hosts may reach Administrative hosts, and Administrative hosts have the most restricted access. First matching rule wins.

\section{Methodology}

\subsection{IP Hopping Mechanism}

Rushyendra designed the core hopping logic, summarized in Algorithm~\ref{alg:iphopping}. When a zone's timer fires, the scheduler walks through each host in that zone, picks a fresh IP at random from the unused portion of the pool, tears down the old flow rules, installs new ones, updates DNS, and logs the event.

\begin{algorithm}[htbp]
\caption{Zone-Based IP Hopping Procedure}
\label{alg:iphopping}
\begin{algorithmic}[1]
\Require Zone configuration $Z = \{z_1, z_2, z_3\}$, IP pool $P$, interval map $I$
\Ensure Updated NAT mappings and flow rules
\For{each zone $z_i \in Z$}
    \If{$elapsed\_time(z_i) \geq I(z_i)$}
        \For{each host $h_j \in z_i$}
            \State $ip_{old} \gets \text{NAT}[h_j.\text{private\_ip}]$
            \State $ip_{new} \gets \text{RandomSelect}(P \setminus \text{ActiveIPs})$
            \State \text{DeleteFlows}($h_j$, $ip_{old}$)
            \State \text{InstallNATFlows}($h_j$, $ip_{new}$)
            \State \text{NAT}[$h_j.\text{private\_ip}$] $\gets ip_{new}$
            \State \text{UpdateDNS}($h_j.\text{hostname}$, $ip_{new}$)
            \State \text{LogShuffle}($h_j$, $ip_{old}$, $ip_{new}$)
        \EndFor
        \State \text{ResetTimer}($z_i$)
    \EndIf
\EndFor
\end{algorithmic}
\end{algorithm}

At the OpenFlow level, the NAT works like this:

\begin{verbatim}
# Outbound: private_src -> public_src
match = OFPMatch(eth_type=0x0800,
                 ipv4_src=private_ip)
actions = [OFPActionSetField(
               ipv4_src=public_ip),
           OFPActionOutput(out_port)]

# Inbound: public_dst -> private_dst
match = OFPMatch(eth_type=0x0800,
                 ipv4_dst=public_ip)
actions = [OFPActionSetField(
               ipv4_dst=private_ip),
           OFPActionOutput(in_port)]
\end{verbatim}

To avoid dropped packets during the switch-over, Rushyendra and Kishan added a short \textit{shadow period} in which both the old and new rules are active simultaneously. Once the new rule is confirmed, the old one is removed.

\subsection{Threat Model}

Siddardha et al.\ defined the attacker we defend against. We assume an adversary who can:

\begin{itemize}
    \item \textbf{Scan the network} with tools like Nmap to discover live hosts and open ports.
    \item \textbf{Move laterally} from a compromised machine toward higher-value targets such as EHR servers.
    \item \textbf{Spoof IP addresses} to impersonate trusted devices and slip past basic access controls.
    \item \textbf{Deploy ransomware} once enough of the network layout is known.
\end{itemize}

We assume the attacker has no access to the SDN controller or to the private 10.0.0.0/24 address space. The goal of the defense is to make reconnaissance data go stale quickly, raise the cost of lateral movement, and shrink the window of opportunity for each host.

\subsection{Security Metrics}

Rushyendra and Siddardha chose four metrics to measure the system's effectiveness:

\begin{enumerate}
    \item \textbf{Reconnaissance Accuracy ($R_A$)}: What fraction of hosts does a scan correctly identify?
    \begin{equation}
        R_A = \frac{|\text{Hosts discovered with valid IPs}|}{|\text{Total active hosts}|}
    \end{equation}

    \item \textbf{Attack Effort Multiplier ($E_M$)}: How many more scans does the attacker need compared to a static network?
    \begin{equation}
        E_M = \frac{\text{Scans required under MTD}}{\text{Scans required under static}} = \frac{T_{hop}^{-1} \cdot T_{scan}}{1}
    \end{equation}
    Here $T_{hop}$ is the rotation interval and $T_{scan}$ is the time one full scan takes.

    \item \textbf{Packet Delivery Ratio ($PDR$)}: How many packets get through during a shuffle?
    \begin{equation}
        PDR = \frac{\text{Packets received}}{\text{Packets sent}} \times 100\%
    \end{equation}

    \item \textbf{Flow Update Latency ($\delta_f$)}: How long does it take from the moment we trigger a hop until the new flow rules are live on the switch?
\end{enumerate}

\section{Implementation}

Rushyendra and Kishan built the system with the following stack:

\begin{itemize}
    \item \textbf{Controller}: Python~3.8, Ryu SDN framework, eventlet~0.30.2, OpenFlow~1.3. The whole thing runs in a Docker container so anyone can reproduce it.
    \item \textbf{Network}: Mininet~2.3.0 with Open vSwitch~2.13 creates the virtual hospital LAN.
    \item \textbf{Host agents}: A tiny Python HTTP server on each host (port~8080) receives secure payloads and writes acknowledgment logs.
    \item \textbf{Scheduler}: A standalone Python module that hits the controller's REST API whenever a zone's timer expires.
    \item \textbf{Dashboard}: An HTML/JS page that polls \texttt{/status} and \texttt{/logs} every few seconds to show live NAT bindings and rotation history.
    \item \textbf{Validation tools}: Nmap for scanning, Wireshark for packet capture, and cURL for triggering secure transfers.
\end{itemize}

Thread safety matters because the controller, the scheduler, and the REST server all touch the same data structures. We protect shared state with mutex locks and write every change to SQLite (tables: \texttt{hosts}, \texttt{nat\_mappings}, \texttt{shuffle\_log}, \texttt{dns\_records}).

Kishan and Jeeshitha handled communication testing. We deliberately chose the \texttt{/sim/secure\_transfer} REST endpoint over \texttt{pingall} for validation, because NAT rewrites can make ICMP replies look wrong even when real data is flowing fine.

\section{Experimental Setup}

\subsection{Testbed}

Our testbed consists of:

\begin{itemize}
    \item Six Mininet hosts ($h_1$--$h_6$), one Open vSwitch, one Ryu controller.
    \item Zone assignments: $h_1, h_2$ (Critical), $h_3, h_4$ (Imaging), $h_5, h_6$ (Administrative).
    \item Private IPs: 10.0.0.1--10.0.0.6 (never change).
    \item Public pool: 172.16.0.50--172.16.0.99 (50 addresses to draw from).
    \item The controller runs in Docker with \texttt{--network host} so it can reach Mininet on the same machine.
\end{itemize}

At first glance, giving the highest-risk zone the longest interval seems backwards. But in practice, we found that rotating Critical-zone IPs too fast caused brief sync delays between the controller and host agents---exactly the kind of hiccup you cannot afford on a ventilator network. Longer intervals for Critical hosts keep connections rock-solid while still providing meaningful defense. Administrative hosts, where a momentary blip is harmless, rotate the fastest.

\subsection{Attack Scenarios}

Rushyendra et al.\ designed three test scenarios:

\begin{enumerate}
    \item \textbf{Reconnaissance}: Repeated Nmap scans (\texttt{nmap -sn 172.16.0.0/24}) to see how fast the attacker's host map goes stale.
    \item \textbf{Lateral movement}: An attacker on an Administrative host tries to reach a Critical host, but the target's IP keeps changing between the scan step and the exploit step.
    \item \textbf{Communication stability}: Continuous REST-based secure transfers between host pairs while IP rotation is running, to make sure legitimate traffic survives.
\end{enumerate}

\textit{Assumptions}: (i)~The attacker cannot reach the controller or the private subnet. (ii)~A single scan takes roughly as long as one rotation interval. (iii)~Mininet's forwarding behavior is representative of a real OpenFlow switch. (iv)~A single-switch topology gives a lower bound on latency; multi-switch deployments would add some overhead.

\section{Results and Evaluation}

\subsection{Reconnaissance Gets Much Harder}

Rushyendra and Kishan ran 20 consecutive Nmap scans, each lasting 15--20~seconds, against both the static baseline and the MTD-enabled network.

In the static case, every scan found all six hosts---100\% accuracy, every time. With MTD-HealthNet running, the picture changed quickly:

\begin{itemize}
    \item \textbf{First scan}: 6/6 hosts found (rotation has not kicked in yet).
    \item \textbf{Scans 2--5}: About 3.2 out of 6 found on average, as Administrative IPs start rotating.
    \item \textbf{Scans 6--20}: Only about 1 host found per scan once all zones have rotated at least once.
\end{itemize}

Steady-state accuracy settled at roughly \textbf{16.7\%}---an \textbf{83.3\% drop} from the static baseline. The one host that still shows up corresponds to whatever address was just freshly assigned and has not yet rotated away.

\subsection{Attackers Must Work Much Harder}

In a static network, one scan is enough to map everything. With rotation, the attacker has to keep rescanning before the map expires. With an 80-second Administrative-zone interval and an 18-second scan:

\begin{equation}
    E_M = \frac{80}{18} \approx 4.4\times \text{ (Administrative zone)}
\end{equation}

For the 120-second Critical zone:

\begin{equation}
    E_M = \frac{120}{18} \approx 6.7\times \text{ (Critical zone)}
\end{equation}

Averaged across all zones, the effort multiplier is \textbf{4.7$\times$}. In plain terms, the attacker spends nearly five times as long scanning for the same amount of useful intelligence.

\subsection{Lateral Movement Mostly Fails}

Kishan ran 15 lateral-movement trials. In each one, the attacker scanned from an Administrative host, identified a Critical-zone target, waited briefly, then tried to connect.

\begin{itemize}
    \item \textbf{Static network}: 15/15 succeeded (100\%).
    \item \textbf{MTD-HealthNet}: 3/15 succeeded (\textbf{20\%}).
\end{itemize}

The 80\% failure rate comes from a simple cause: the target's IP changed between the scan and the connection attempt. On top of that, the ACL rules block Administrative$\rightarrow$Critical traffic outright, adding a second layer of protection.

Rushyendra et al.\ noted the following details:

\begin{itemize}
    \item The \textbf{2\% PDR dip} only happens in the brief window while flow rules are being swapped. The shadow period keeps it small.
    \item Steady-state \textbf{RTT rises by just 0.07~ms} due to NAT header rewriting. During a hop, RTT can spike to 2.8~ms for a second or two before settling back down.
    \item \textbf{12.4~ms flow update latency} is fast enough that no healthcare application would notice---most clinical protocols tolerate latency variations on the order of seconds, not milliseconds.
    \item Every application-layer secure transfer we attempted succeeded with a \textbf{100\% delivery rate}, even mid-rotation. This confirms that the NAT layer keeps real communication intact.
\end{itemize}

\section{Case Studies}

\subsection{Case 1: Ransomware Aimed at an EHR Server}

\textit{Setup}: The attacker takes over an Administrative workstation ($h_5$) and wants to push ransomware to the Critical-zone EHR server ($h_1$).

\textit{In a static network}: The attacker scans, finds $h_1$ at a fixed IP, and delivers the payload. Straightforward.

\textit{With MTD-HealthNet}: The attacker scans and sees $h_1$ at, say, 172.16.0.57. Crafting the exploit takes 60--90~seconds. By then $h_1$'s IP has rotated (120\,s interval), so the payload hits a dead address. Even if the attacker rescans, the ACL blocks Administrative$\rightarrow$Critical traffic anyway. Two independent defenses must both fail for the attack to succeed.

\subsection{Case 2: Lateral Movement Through the Imaging Zone}

\textit{Setup}: The attacker is on $h_3$ (Imaging) and wants to reach $h_4$ to steal DICOM data.

\textit{With MTD-HealthNet}: The attacker sees $h_4$ at 172.16.0.62. After 100~seconds, $h_4$ moves to 172.16.0.78. The connection to the old address fails silently. By the time the attacker scans again and finds the new IP, another rotation may already be underway. The attacker gets caught in a loop of stale information.

\subsection{Case 3: Persistent Network Scanning}

\textit{Setup}: An outsider runs continuous Nmap scans to build a complete network map.

\textit{With MTD-HealthNet}: The first scan gets all six hosts. Eighty seconds later, two Administrative IPs have changed. After 100~seconds, Imaging IPs rotate too. By 120~seconds, every zone has shuffled at least once. The attacker's map is now a patchwork of current and expired entries, with only about 1~in~6 entries still valid. Worse, all visible IPs come from the 172.16.0.0/16 pool, revealing nothing about the real 10.0.0.0/24 topology underneath.

\section{Discussion}

\subsection{What Works Well}

\begin{itemize}
    \item \textbf{Stale intelligence}: Any scan result expires within one rotation interval, forcing the attacker to start over.
    \item \textbf{Hidden internals}: Even a successful scan only reveals the public pool; the private layout stays secret.
    \item \textbf{Tunable per zone}: Administrators can tighten rotation for risky zones and relax it for stable ones, matching each department's tolerance for disruption.
    \item \textbf{Real data survives}: We proved this at the application layer, not just with ping.
    \item \textbf{Full audit trail}: Every IP change is logged in SQLite, supporting the kind of traceability HIPAA demands.
\end{itemize}

\subsection{Limitations}

\begin{itemize}
    \item \textbf{Single controller}: If the Ryu process crashes, all MTD activity and NAT translations stop until it restarts.
    \item \textbf{One switch}: Our testbed uses a single OVS instance, which is simpler than a real hospital's multi-switch fabric.
    \item \textbf{Legacy devices}: Equipment that hardcodes peer IPs will need a proxy layer we have not yet built.
    \item \textbf{Address-space entropy}: With 50~pool addresses and 6~hosts, each rotation offers about $\log_2\binom{50}{6} \approx 22.7$ bits of randomness---decent, but not cryptographic.
    \item \textbf{Minimum interval}: Below roughly 30~seconds, sync delays between the controller and host agents become noticeable.
\end{itemize}

\subsection{Is the Trade-off Worth It?}

Rushyendra and Jeeshitha weighed the costs against the gains. Giving up 2\% of packet delivery and adding 0.07~ms of steady-state latency buys an 83.3\% cut in reconnaissance accuracy and an 80\% cut in lateral-movement success. In a hospital, where clinical protocols easily tolerate millisecond-level jitter, this is a favorable deal.

The zone-based design makes the trade-off adjustable. If a particular ward needs rock-solid connectivity, its rotation interval can be lengthened. If a low-risk subnet can handle more churn, its interval can be shortened to harden it further.

\section{Conclusion}

MTD-HealthNet, developed by Rushyendra et al., shows that a hospital network does not have to be a sitting target. By shuffling public-facing IPs on a zone-aware schedule and hiding the real topology behind NAT, the system cuts reconnaissance accuracy by 83.3\%, raises attacker effort by 4.7$\times$, and blocks 80\% of lateral-movement attempts---all while keeping packet delivery above 97.8\% and adding negligible latency in normal operation.

The prototype confirms that dynamic address rotation is practical for healthcare environments. It requires no new hardware, imposes minimal overhead, and gives administrators per-zone control over the security--stability balance.

\section{Future Work}

Jeeshitha and Siddardha outline several next steps:

\begin{itemize}
    \item \textbf{Adaptive intervals}: Use anomaly-detection models to speed up rotation when a threat is detected and slow it down when the network is calm, replacing fixed timers with intelligent ones.
    \item \textbf{IDS integration}: Let the intrusion detection system trigger an immediate shuffle for any host it flags, adding a reactive layer on top of the proactive schedule.
    \item \textbf{Scaling up}: Test on multi-switch, multi-subnet topologies and measure how controller CPU and memory usage grow with host count.
    \item \textbf{Port randomization}: Extend the shuffle to transport-layer ports, rewriting them transparently so that applications stay unaware.
    \item \textbf{Formal analysis}: Apply game-theoretic models to compute provably optimal rotation intervals as a function of attacker scan speed and network size.
    \item \textbf{Real-world pilot}: Deploy on a physical hospital testbed to validate under realistic traffic, latency constraints, and regulatory requirements.
\end{itemize}
